## ความจริงข้อแรก: AI คุยกันตรงๆ ไม่ได้

แต่ละทีม (Manus / Codex / Claude) เป็น AI คนละตัว คนละ session คนละ repo
**ไม่มีใครส่งข้อความหากันเองได้** การเชื่อมงานเกิดผ่าน 3 ช่องทางเท่านั้น:

| # | ช่องทาง | ใช้เมื่อ | ระดับ |
|---|---------|---------|-------|
| 1 | **Supabase (ข้อมูลกลาง)** | เชื่อมข้อมูลปกติทุกวัน | อัตโนมัติ ไม่ต้องนัด |
| 2 | **repo นี้ (contracts + CHANGELOG + Issues)** | จะเปลี่ยนโครงสร้าง/key/API | ต้องนัดแนะ |
| 3 | **เจ้าของ (numjebar)** | ตัดสินใจเรื่องใหญ่ / ทิศทาง | คนตัดสิน |

## 🔒 กฎเหล็ก (ห้ามข้าม)

### กฎ 1 — Contract First
ห้ามเปลี่ยนโครงสร้างข้อมูลกลาง / key / API โดยแก้แค่ในโค้ดตัวเอง
**ต้องแก้ที่ repo นี้ก่อน** แล้วทีมอื่นถึงจะรู้และปรับตามได้

### กฎ 2 — ใครจะเปลี่ยน contract ต้องทำ 3 อย่างนี้เสมอ
1. เปิด **GitHub Issue** ใน repo นี้ อธิบายว่าจะเปลี่ยนอะไร เพราะอะไร กระทบทีมไหน
2. แก้ไฟล์ `DATA_CONTRACT.md` ให้ตรงกับของจริง
3. เพิ่มบรรทัดใน `CHANGELOG.md` (วันที่ · ทีม · เปลี่ยนอะไร · ทีมที่กระทบ)

### กฎ 3 — Backward compatible ไว้ก่อน
เพิ่ม field ใหม่ได้เสมอ แต่ **ห้ามลบ/เปลี่ยนชื่อ/เปลี่ยนความหมาย field เดิม**
โดยไม่ประกาศล่วงหน้า เพราะทีมอื่นอาจใช้อยู่ → จะทำให้แอปเขาพัง

### กฎ 4 — Key กลางต้องตรงกัน
`db.ingredients[].id`, `db.menus[].id` คือกุญแจเชื่อมทุกระบบ ห้ามแก้รูปแบบเด็ดขาด
(ดูตารางเต็มใน `DATA_CONTRACT.md`)

### กฎ 5 — เขียนเฉพาะ zone ของตัวเอง
- **Manus / Codex / Claude** เขียนกลับได้เฉพาะ **stock / operation zone**
- **ห้ามทับ** master data ของ JEBAR DATA (menus · สูตร · ingredients · รูปหลัก)

### กฎ 6 — ห้าม secret ลง repo
❌ ห้าม commit API key / anon key / token ใด ๆ — ใช้ environment / secret store เท่านั้น

## เวิร์กโฟลว์เวลาต้องเปลี่ยนข้อมูลกลาง (ตัวอย่าง)

```
Manus ต้องการ field ใหม่ "menu.tags" สำหรับหน้า Landing
   │
   ├─1. เปิด Issue ใน jebar-contracts: "ขอเพิ่ม db.menus[].tags"
   │      → tag ทีมที่เกี่ยว, รอ ack จากเจ้าของ/Claude
   │
   ├─2. ตกลงแล้ว → แก้ DATA_CONTRACT.md เพิ่ม field tags
   │      → เพิ่มบรรทัดใน CHANGELOG.md
   │
   └─3. แต่ละทีมอ่าน CHANGELOG → ปรับโค้ดฝั่งตัวเอง → ปิด Issue
```
